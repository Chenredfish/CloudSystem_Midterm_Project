# Container 期中專題 — 分散式共享帳本

## 專題目標

透過 Docker 容器技術，實作一個簡易的**分散式帳本系統**，具備基本區塊鏈概念，包含交易紀錄、區塊串接與完整性驗證，並模擬多個 Client 同步操作帳本的情境。

---

## 系統功能需求

| 功能 | 程式 | 說明 |
|------|------|------|
| 查詢帳戶餘額 | `app_checkMoney.py <帳戶>` | 查詢指定帳戶目前的餘額 |
| 查詢交易紀錄 | `app_checkLog.py <帳戶>` | 列出指定帳戶的所有交易歷史 |
| 進行轉帳 | `app_transaction.py <來源> <目標> <金額>` | 帳戶間金額轉移，記錄於區塊中 |
| 驗證帳本完整性 | `app_checkChain.py` | 檢查所有區塊的 SHA256 串接是否正確 |

---

## 系統設計規範

- 可使用任意程式語言開發
- 所有帳本資料儲存在共用資料夾（`/share` 或 `/storage`）
- 每 **5 筆交易**組成一個區塊（Block），以文字檔儲存（`1.txt`, `2.txt`, ...）
- 每個區塊包含：
  - 前一區塊的 SHA256 雜湊值
  - 下一區塊的連結資訊
  - 該區塊的所有交易紀錄（格式：`來源, 目標, 金額`）
- 區塊達到 5 筆交易上限時，系統**自動建立新區塊**並計算前一區塊 SHA256
- `app_checkChain.py` 驗證完整後，執行者可從 `angel` 得到 **10 元獎勵**

### 區塊格式範例

```
Sha256 of previous block: <hash>
Next block: 2.txt
angel, A, 100
angel, B, 100
angel, C, 100
A, B, 10
A, C, 20
```

## 系統架構

```
Client 1          Client 2          Client 3
   |                 |                 |
/storage          /storage          /storage
(共用帳本區塊 1~N，三個節點同步)
```

3 個 Docker Container 共享同一份帳本資料夾。

#### 若選 C1-a（分散式同步）

**C1a-1. 帳本要「真的分散」還是「假裝分散」？**
這是做這個主題最核心的問題：要讓助教信服「帳本真的放在每個節點」，資料就不能共用同一個資料夾。
- [V] a. **真分散** — 每個 container 掛自己獨立的 volume（`client1_data`、`client2_data`、`client3_data`），寫入時用 HTTP 推送給其他節點；**展示效果最強，能清楚展示「同步」這個動作，但要多寫一個 HTTP server 常駐在每個 container** 註：希望之後特別說明如何達成
- [ ] b. **共享 volume + 本地副本** — 仍然共用同一個資料夾，但每個節點另外存一份自己的 copy；**做起來簡單，但其實帳本天生就同步了，「分散式同步」的概念被稀釋，助教可能會問「你這樣算哪門子分散式？」**
- [ ] c. ___________

**C1a-2. 節點間怎麼傳資料？**
選 a（真分散）之後，寫入的節點要怎麼通知其他兩個節點？
- [V] a. 在每個 container 常駐一個 Flask HTTP server — 寫入後 `POST /sync` 把新的區塊內容發給其他兩個節點；**寫法直觀，curl 就能測試，demo 時可以直接在瀏覽器或 terminal 展示 API call** 
- [ ] b. 用 Python `socket` 自建 TCP 協議 — 自己定義傳輸格式；**比 HTTP 複雜，沒有現成 framework，容易出奇怪的 bug，除非有特別理由否則不推**
- [ ] c. ___________

**C1a-3. 兩個節點幾乎同時寫入時，怎麼決定誰的資料算數？**
（Demo 時手動操作大概不會真的撞到，但助教可能會問）
- [V] a. 先到先得 — 第一個拿到鎖的節點寫入，後來的節點收到「已有新交易」的回應後重試或等待；**邏輯清楚，符合大多數帳本系統的直覺** 註：也希望未來特別標註此設計
- [ ] b. 最後一筆贏 — 誰最後寫進去就算誰的；**簡單，但兩筆交易都會發生，只是順序不確定，帳本一致性較弱**
